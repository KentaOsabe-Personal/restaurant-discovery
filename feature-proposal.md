# 機能開発提案: Restaurant Discovery 次期開発ロードマップ

## 前提

| 項目 | 内容 |
|---|---|
| 利用者 | 個人（自分専用） |
| 主な検索エリア | 新潟市内 |
| メイン利用シーン | **夜の店探し**（飲み会・ディナー） |
| 不要な機能 | ブックマーク・訪問記録（食べログで管理） |
| コスト方針 | Google Places APIコストはなるべく抑えたい |
| 不要な機能2 | 地図表示（Google Mapsリンクで十分） |

## 現状の課題

現在のアプリは「自然文入力 → AI解析 → Google Places検索 → AI推薦3-5件」の基本パイプラインが動作するが、以下の点で日常使いにはまだ遠い。

1. **毎回テキストを打つのが面倒** — クイックアクセス手段がない
2. **結果がテキストだけ** — 写真も営業時間もなく、店の雰囲気がわからない
3. **食べログとの連携がない** — 気になった店を食べログで確認するのに手動検索が必要
4. **検索条件が不透明** — AIが何をどう解釈したか見えない
5. **3-5件しか見えない** — もっと候補を見たい時に手段がない

---

## Phase 1: 毎日使える基盤

> APIコスト増: **なし** / フロントエンド中心の改善

### 1. クイック検索ボタン

**概要**
検索バーの下に、新潟の夜の店探しに最適化されたプリセットボタンを配置。ワンタップで検索を実行できる。

**ボタン例**
- エリア: `万代` `古町` `駅前` `駅南` `東中通`
- シーン: `飲み会` `デート` `一人飲み` `二次会` `接待`
- ジャンル: `居酒屋` `焼肉` `寿司` `ラーメン` `バー`
- 組み合わせ: `古町で飲み会` `万代でデート` `駅前で一人飲み`

タップすると検索バーにクエリが入り、即検索実行。複数ボタンの組み合わせも可能（エリア + シーンなど）。

**価値**
夜の店探しで最もよくある検索パターンをワンタップ化。テキスト入力の手間を大幅削減。

**技術アプローチ**
- フロントエンドのみ（新コンポーネント `QuickSearchButtons`）
- ボタンクリック時に `onSubmit` にクエリ文字列を渡すだけ
- ボタンの定義は定数配列で管理

**複雑度**: 小 / **APIコスト影響**: なし（通常の検索1回分）

---

### 2. 食べログリンク自動生成

**概要**
各PlaceCardに「食べログで見る」リンクを追加。店名で食べログを検索するURLを自動生成する。

**リンク先URL**
```
https://tabelog.com/niigata/rstLst/?vs=1&sk={エンコード済み店名}
```

**価値**
気になった店を食べログで確認する既存ワークフロー（口コミ確認 → ブックマーク）をワンクリックでつなげる。現状の「店名をコピー → 食べログを開く → 検索」の3ステップが1ステップに。

**技術アプローチ**
- `PlaceCard` コンポーネントに食べログリンクを追加
- `encodeURIComponent(name)` でURL生成
- 新潟県（`/niigata/`）固定で精度向上

**複雑度**: 小 / **APIコスト影響**: なし

---

### 3. 検索履歴

**概要**
直近の検索クエリを最大10件保存し、検索バー付近に表示。タップで即再検索。

**UI**
- 検索バーの下にチップ形式で表示（例: `古町 居酒屋` `万代 焼肉`）
- 各チップにバツボタンで個別削除可能
- 「履歴クリア」ボタン

**価値**
「先週良かったあの検索をもう一度」がワンタップ。同じ条件での再検索が非常に楽になる。

**技術アプローチ**
- `localStorage` で保存（バックエンド不要）
- カスタムフック `useSearchHistory` で管理
- 検索実行時に自動保存、重複排除、FIFO（10件超えたら古いものを削除）

**複雑度**: 小 / **APIコスト影響**: なし

---

### 4. 「おまかせ」ボタン

**概要**
「今夜どこ行く？迷ったらこれ！」的なボタン。押すと新潟市内の夜向けおすすめをランダムに提案する。

**動作**
- ボタンを押すと「新潟市で今夜おすすめの店」のようなクエリで検索を実行
- ジャンルやエリアをランダムに変えることで毎回違う結果に
- プリセットクエリの配列からランダム選択:
  - `"新潟市 今夜のおすすめ居酒屋"`
  - `"古町 隠れ家的な店"`
  - `"万代 コスパの良い飲み屋"`
  - `"新潟駅前 〆のラーメン"` など

**価値**
「何食べたいかわからない」問題を解決。アプリ名「今日どこ行く？」の体験そのもの。

**技術アプローチ**
- フロントエンドのみ（プリセットクエリ配列からランダム選択 → 既存検索パイプライン実行）
- 新コンポーネント or SearchInputに「おまかせ」ボタン追加

**複雑度**: 小 / **APIコスト影響**: 通常検索1回分

## Phase 2: 検索結果の充実化

> コスト効率を重視した段階的な情報追加

### 5. 解析条件タグ表示

**概要**
検索後、AIがクエリから解析した条件（エリア・ジャンル・価格帯）をタグとして検索バーの下に表示する。

**UI例**
```
検索: "古町で安くてうまい焼肉"
→ [エリア: 古町] [ジャンル: 焼肉] [価格帯: ¥]
```

**価値**
- AIが検索意図を正しく理解しているか確認できる
- 「あれ、エリア違う」と気づいたら再検索の判断材料になる
- 検索体験の透明性向上

**技術アプローチ**
- **バックエンド変更なし** — 既にレスポンスに `parsed_conditions` が含まれている
- フロントエンドで `parsed_conditions` を受け取りタグ表示する新コンポーネント `ParsedConditionTags`
- `App.tsx` の state に `parsedConditions` を追加

**複雑度**: 小 / **APIコスト影響**: なし

---

### 6. 「もっと見る」機能

**概要**
AI推薦の3-5件以外に、Google Placesから取得した残りの候補も閲覧可能にする。

**動作**
- 初期表示: AI推薦3-5件（現状通り）
- 「もっと見る」ボタンを押すと残りの候補を追加表示
- 追加候補にはAI推薦理由はないが、評価・価格帯・住所は表示
- AI推薦分と視覚的に区別（例: 推薦カードには「AIおすすめ」バッジ）

**価値**
「AIのおすすめ以外も気になる」時に、追加検索なしで候補を広げられる。

**技術アプローチ**
- バックエンドのレスポンスに `all_places`（全20件）を追加
- フロントエンドで推薦分とそれ以外を分離表示
- `SearchController` で `places` をレスポンスに含める
- `PlaceCard` に `reason` なし時の表示パターン追加

**複雑度**: 小〜中 / **APIコスト影響**: なし（既に取得済みの20件を活用）

## Phase 3: スマート機能

> AIの能力を活かした体験の高度化

### 7. 時間帯対応検索

**概要**
検索時の現在時刻に応じて、自動的に適切な文脈をクエリに付加する。

**ルール**
| 時間帯 | 付加コンテキスト |
|---|---|
| 11:00-14:00 | ランチ営業 |
| 17:00-22:00 | ディナー・居酒屋 |
| 22:00-02:00 | 深夜営業・〆の店 |
| その他 | カフェ・モーニング |

**例**
- 19:00に「古町 焼肉」→ AIが「古町で夜に行ける焼肉」として解釈
- 23:00に「ラーメン」→「新潟で深夜営業のラーメン」として処理

**価値**
「今」行ける店を自動的に絞り込む。夜メインの利用なら、明示的に「夜」と入れなくても夜向けの結果が出る。

**技術アプローチ**
- フロントエンドで現在時刻を取得 → APIリクエストに `context_time` を追加
- `QueryParserService` のプロンプトに時間帯コンテキストを追加
- 既存パイプラインへの変更は最小限（プロンプト調整のみ）

**複雑度**: 中 / **APIコスト影響**: なし

---

### 8. 人数・予算クイック入力

**概要**
飲み会幹事向けに、人数と一人あたり予算を簡単に指定できるUIを追加。

**UI**
- 検索バーの下に折りたたみ式の詳細条件パネル
- 人数: `2人` `3-4人` `5人以上` ボタン
- 予算/人: `~3,000円` `~5,000円` `~8,000円` `8,000円~` ボタン
- 選択した条件はクエリに自動付加

**例**
- クエリ「古町 居酒屋」+ 人数「5人以上」+ 予算「~5,000円」
  → 「古町で5人以上入れる一人5000円以下の居酒屋」として検索

**価値**
飲み会の幹事として店を探す時、人数と予算は必ず考える条件。テキストで書くより選ぶ方が楽。

**技術アプローチ**
- 新コンポーネント `DetailedConditions`（折りたたみパネル）
- 選択値をクエリ文字列に結合してから既存パイプラインに投入
- バックエンド変更なし（自然文に条件を組み込むだけ）

**複雑度**: 中 / **APIコスト影響**: なし

## 実装優先度まとめ

| 優先度 | 機能 | 複雑度 | APIコスト | 価値 |
|---|---|---|---|---|
| **1** | クイック検索ボタン | 小 | なし | 毎日の利用で最も効く |
| **2** | 食べログリンク | 小 | なし | 既存ワークフローとの橋渡し |
| **3** | 検索履歴 | 小 | なし | リピート検索の効率化 |
| **4** | 「おまかせ」ボタン | 小 | なし | 迷った時の体験 |
| **5** | 解析条件タグ | 小 | なし | 検索の透明性 |
| **6** | もっと見る | 小〜中 | なし | 候補の幅を広げる |
| **7** | 時間帯対応 | 中 | なし | 自動的に適切な結果 |
| **8** | 人数・予算入力 | 中 | なし | 幹事ユースケース対応 |

全機能においてAPIコスト増なし。Phase 1の4機能はすべてフロントエンドのみで実装でき、即効性が高い。ここから着手するのが最も効率的。
