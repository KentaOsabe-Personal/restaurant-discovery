# Roadmap

## 概要

既存の居酒屋・バー検索に加え、ラーメン屋検索モードを新設する。
画面上部にタブを設置し、2つの検索体験を切り替えられるようにする。
既存の検索パイプライン（QueryParser → GooglePlaces → Recommendation）をモードパラメータで拡張し、ラーメン固有のプロンプト・検索パラメータに対応する。

## アプローチ決定

- **選択**: 既存パイプライン拡張（モードパラメータ方式）
- **理由**: コード重複なし、既存パイプラインの実績を活かせる、個人利用アプリに適したシンプルさ
- **却下した案**:
  - 完全分離パイプライン（新エンドポイント方式）: コード重複が多く、共通ロジック変更時に2箇所修正が必要
  - 戦略パターン（プロファイル方式）: 個人利用アプリに対して抽象化が過剰

## スコープ

- **含む**:
  - タブ切り替えUI（居酒屋・バー / ラーメン）
  - ラーメン検索（自然文 + ジャンル自動付与 + AI推薦）
  - 自宅起点の距離フィルター（車N分圏内）
  - ラーメンおまかせ機能（距離ベース + エリアランダム選定）
  - 検索履歴のタブ別管理
  - 食べログリンク・フィードバック再レコメンドのラーメンタブ対応
- **含まない**:
  - つけ麺・まぜそば等のサブジャンル別タブ
  - ラーメン店の口コミ・評価機能
  - Google Maps Directions API による正確な移動時間算出
  - ラーメン以外の新モード（カフェ等）

## 制約

- 完全個人利用（PC上Docker起動）
- 居酒屋は繁華街中心（新潟市内・長岡）
- ラーメンは車移動前提 → 繁華街に限定しない、距離ベース検索
- 自宅の緯度経度はconfigファイルで管理
- 距離計算は直線距離の概算で十分（Haversine公式）

## バウンダリ戦略

- **この分割の理由**: 各フェーズが独立してリリース可能な価値単位。Phase 1でラーメン検索が動き、Phase 2で距離フィルターが加わり、Phase 3でおまかせが完成する。
- **注意すべき共有境界**:
  - `/api/search` エンドポイントの `mode` パラメータが全Specの共通接点
  - QueryParserService / RecommendationService のプロンプト切り替えロジック
  - フロントエンドの `App.tsx` タブ状態管理が各Specのベース

## Specs (dependency order)

- [ ] ramen-search-mode -- タブUI + ラーメン検索基本形（フロントエンド＆バックエンド一気通貫）. Dependencies: none
- [ ] distance-filter -- 自宅位置config + 距離フィルターUI + バックエンドロジック. Dependencies: ramen-search-mode
- [ ] ramen-omakase -- ラーメンおまかせ機能（距離ベース + エリアランダム選定）. Dependencies: ramen-search-mode, distance-filter
